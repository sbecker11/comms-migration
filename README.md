# comms-migration

Tools and rules for consolidating incoming communications (email, and over time
voice/SMS) into two hubs — **Professional** and **Personal** — and routing every
sender to the correct one.

This repo is the single source of truth for **routing**: where mail goes. It is
deliberately separate from the job-search automation that *processes* recruiting
mail once it lands.

---

## ⚠️ Real data never gets committed

Some files in this project are generated from, or contain, real personal data
(names, emails, phone numbers for hundreds of contacts). **These are gitignored.
Only synthetic `.example` stubs are committed.**

| Real file (gitignored)   | Committed stub                  | Generated by                  |
|--------------------------|---------------------------------|-------------------------------|
| `contacts/Contacts.yaml` | `contacts/Contacts.yaml.example`| edited via the contacts app   |
| `rules/senders.yaml`     | `rules/senders.yaml.example`    | `scripts/export_senders_cli.py`   |

Rules of the road:

- **Never** `git add -f` either real file. The `.gitignore` entries are the only
  thing standing between your address book and a public commit.
- Any **new file derived from** `Contacts.yaml` inherits the same data and must be
  gitignored too. The source being ignored does not protect its exports.
- When you change a schema, update the matching `.example` stub so it stays an
  accurate, committable reference.

A `pre-commit` hook is included that blocks staged secrets and junk files. Note it
does **not** flag the personal-data files (they contain no credentials) — the
`.gitignore` lines are their protection. See [Setup](#setup).

## The two-hub model

Every sender resolves to exactly one hub:

- **Professional** — business contacts, clients, recruiters, vendors.
- **Personal** — everyone else (default).

Contacts carry hub assignments as groups (`Professional` / `Personal`), plus an
`Is Deleted` group for soft deletion. A contact with no hub assignment defaults to
**Personal** (see `scripts/default_contacts_personal.py`).

`rules/senders.yaml` compiles those assignments into a routing table the classifier
can apply, matched in order:

```
exact email  →  exact phone  →  domain (longest match wins)  →  overrides  →  default
```

## The `rules/` layer model

Routing and classification are split into three layers. The committable rule is
simple: **commit the policy, ignore the data.** A file that describes *how*
routing works is safe to commit; a file that contains *who* (real names, emails,
phones, or anything derived from contacts) stays local with an `.example` stub.

| Layer | File | Contains | Hand-authored or generated | Committed? |
|-------|------|----------|----------------------------|------------|
| 1 — Data | `contacts/Contacts.yaml` | The actual address book: who exists, their emails/phones, hub assignment | Edited via the contacts app | **No — gitignored** (stub: `Contacts.yaml.example`) |
| 2 — Routing | `rules/senders.yaml` | Compiled lookup of which hub each email/phone/domain routes to | Generated from Layer 1 (`export_senders.py`) | **No — gitignored** (stub: `senders.yaml.example`) |
| 3 — Actions | `rules/actions.yaml` | What to *do* per category (notify, tag, digest, quarantine…), trigger signals, action tiers | Hand-authored (mirrors the runbook, Appendix C) | **Yes — committed** |

Why the split falls where it does: Layers 1 and 2 name real people, so they never
enter git. Layer 3 is pure policy — categories and actions, no personal data — so
it is safe to commit and is its own reference (no stub needed). `actions.yaml` is
documented in `comms-migration-runbook.md`.

**Deciding any new `rules/` file:** does it name real people, or only describe
behavior? Names → gitignore it and add an `.example` stub. Behavior only → commit
it directly.

## Setup

```bash
# 1. install the pre-commit hook (blocks staged secrets and junk files)
cp pre-commit .git/hooks/pre-commit && chmod +x .git/hooks/pre-commit

# 2. create your real contacts file from the stub, then populate it
cp contacts/Contacts.yaml.example contacts/Contacts.yaml

# 3. generate the routing table from your contacts
python scripts/export_senders_cli.py
```

`Contacts.yaml` and `senders.yaml` are gitignored, so steps 2–3 stay local.

## Common tasks

```bash
# Regenerate the routing table after editing contacts
python scripts/export_senders_cli.py

# Default any unassigned contacts to the Personal hub
python scripts/default_contacts_personal.py

# Run the contacts web app   (confirm the actual entrypoint/command)
python contacts_app/main.py
```

## Scope boundary

`comms-migration` owns **routing truth** — where communications go. The
**processing** of recruiting mail once it arrives (email classification, JD
evaluation, the ATS resolver, the job tracker) lives in the separate job-search
automation repo, not here. The two meet only at the recruiting-funnel handoff.

See `routing-inventory.md` for the full record of forwards and categorization
routes across both hubs.
